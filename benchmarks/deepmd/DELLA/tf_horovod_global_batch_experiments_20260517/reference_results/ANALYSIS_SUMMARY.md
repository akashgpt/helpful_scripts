## Scope

This analysis combines:

- reused global-batch benchmark runs copied into `runs/update_matched/`;
- new 15-minute training tests in `runs/sample_matched/`, `runs/lr_sensitivity/`, `runs/walltime_matched/`, and `runs/walltime_and_long_baseline/`;
- pseudo-validation by freezing/compressing each benchmark checkpoint and running `dp test` against:
  - `v7_i34/md/*/pre/recal/set_train/deepmd` plus `set_test/deepmd`;
  - `sim_data_ML_v4/v8_i16/md/*/pre/recal/set_train/deepmd` plus `set_test/deepmd`.

The pseudo-validation set is not fully independent: these configurations come from active-learning trajectories generated by a related `pv_comp.pb` model before VASP DFT recalculation. It is still useful as a DFT-labeled stress test for the model family.

## Interpretation

The strongest result is the reused 4GPU/10k update-matched run. It has the best combined pseudo-validation force RMSE and the best energy RMSE/atom, while costing far less GPU time than the 8GPU/10k and 16GPU/10k update-matched runs.

The 8GPU/7k walltime-matched run is the best new large-GPU experiment. It nearly matches the 8GPU/10k and 16GPU/10k update-matched pseudo-validation force RMSE, but still costs about 2.2x the GPU seconds of the 4GPU/10k run.

The sample-matched short runs are useful but not enough to replace update-matched training. The best short sample-matched case, 8GPU/1250 linear, is decent, but worse than 4GPU/10k, 8GPU/7k, and 1GPU/20k.

Learning-rate scaling matters. Linear scaling consistently beats `sqrt` and `none` in the comparable short 4GPU/8GPU runs. The `none` runs are clearly poor.

The 2GPU/5k linear run is an outlier failure by both training loss and pseudo-validation error and should not guide scheduling choices.

## Recommendation

For normal training, do not jump to 8 or 16 GPUs by default. Based on these tests, the practical baseline should be:

1. Use 1GPU for cheap development/debug and long serial baselines.
2. Use 4GPU update-matched training when wall time matters; it gives the best pseudo-validation quality per practical cost in this matrix.
3. Use 8GPU only if shorter wall clock is worth the extra GPU cost, and prefer a schedule like the 8GPU/7k linear walltime-matched run.
4. Avoid 16GPU for routine training unless the schedule is redesigned around global batch size and evaluated by validation metrics, not only training loss.
